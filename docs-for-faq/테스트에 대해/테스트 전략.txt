================================================================================
  관리자 BO 프로젝트 테스트 전략
================================================================================


■ 결론부터
================================================================================

  관리자 CRUD 페이지에서 진짜 중요한 건 UI가 아니라 데이터 정합성이다.
  "저장했는데 데이터가 틀리게 들어감", "조회했는데 빠진 건이 있음"
  이런 게 문제지, 버튼 위치나 레이아웃이 바뀌는 건 문제가 아니다.

  API 서버만 잘 돌아가면 UI는 수정만 하면 된다.
  → 테스트 비중도 API 쪽에 두는 게 맞다.


■ 테스트 종류별 투자 대비 효과
================================================================================

  ┌──────────────────────────────┬────────┬────────┬──────────────────────┐
  │ 테스트 종류                  │ 효과   │ 비용   │ 이 프로젝트 적합도   │
  ├──────────────────────────────┼────────┼────────┼──────────────────────┤
  │ ① API 테스트                │ ★★★   │ 낮음   │ 핵심 — 최우선 투자   │
  │   (Postman / REST Assured)  │        │        │                      │
  ├──────────────────────────────┼────────┼────────┼──────────────────────┤
  │ ② DB 데이터 검증            │ ★★    │ 낮음   │ API 테스트와 병행     │
  ├──────────────────────────────┼────────┼────────┼──────────────────────┤
  │ ③ Playwright (핵심만)       │ ★★    │ 중간   │ 해피패스 5~10개만    │
  ├──────────────────────────────┼────────┼────────┼──────────────────────┤
  │ ④ Playwright (전체 도배)    │ ★     │ 높음   │ 비추 — 유지보수 지옥 │
  ├──────────────────────────────┼────────┼────────┼──────────────────────┤
  │ ⑤ Storybook                │ △     │ 중간   │ 디자인 시스템이면 유용│
  │                              │        │        │ 관리자 BO에서는 장식 │
  └──────────────────────────────┴────────┴────────┴──────────────────────┘


■ ① API 테스트 — 최우선 (가성비 최고)
================================================================================

  왜 핵심인가:
    - 관리자 BO의 가치 = 데이터를 정확하게 CRUD 하는 것
    - API만 검증하면 프론트가 바뀌어도 테스트가 안 깨짐
    - 작성 쉽고, 실행 빠르고, 유지보수 거의 없음

  도구 선택:
  ┌──────────────────┬───────────────────────────────────────────────┐
  │ Postman          │ GUI로 쉽게 작성, 컬렉션 공유 가능             │
  │                  │ Newman으로 CI/CD 자동화 가능                  │
  │                  │ 비개발자도 사용 가능                          │
  ├──────────────────┼───────────────────────────────────────────────┤
  │ REST Assured     │ Java 코드로 작성, JUnit과 통합               │
  │ (Java)           │ 백엔드 개발자에게 친숙                        │
  ├──────────────────┼───────────────────────────────────────────────┤
  │ 직접 fetch/axios │ 프론트 개발자가 JS로 작성                     │
  │ (JS)             │ 별도 도구 없이 가능                          │
  └──────────────────┴───────────────────────────────────────────────┘

  테스트할 것:
    - CRUD 정상 동작 (등록 → 조회 → 수정 → 삭제 → 재조회)
    - 필수값 누락 시 에러 응답
    - 존재하지 않는 데이터 조회/수정/삭제 시 처리
    - 검색 조건별 결과 정확성
    - 응답 포맷 (code, message, data) 일관성

  예시 (매장 스케줄 API):
    1. POST /api/airstar/shopschedulemng/v1.0/saveSchedule
       → body: { storeId: "STORE001", scheduleDate: "20260315", openStatus: "OPEN" }
       → 기대: code=200

    2. POST /api/airstar/shopschedulemng/v1.0/selectScheduleMonthList
       → body: { storeId: "STORE001", searchMonth: "202603" }
       → 기대: data 배열에 위에서 저장한 건이 포함

    3. 필수값 누락 테스트
       → body: { storeId: null }
       → 기대: code=500 또는 validation 에러


■ ② DB 데이터 검증
================================================================================

  API 테스트와 함께 하면 효과적:
    - 저장 API 호출 후 → DB 직접 조회해서 값 일치 확인
    - 특히 매핑 테이블(bs_airstar_store_map) 정합성
    - COALESCE 기본값이 제대로 들어갔는지
    - sync_status, use_yn 기본값 확인


■ ③ Playwright — 해피패스만
================================================================================

  전체 도배가 비효율인 이유:
    - 관리자 BO는 UI가 거의 안 바뀜 → 테스트 가치 낮음
    - 그런데 CSS 클래스 하나, 셀렉터 하나 바뀌면 테스트가 깨짐
    - 깨진 테스트 고치는 데 드는 시간 > QA가 직접 클릭하는 시간

  그래도 만들면 좋은 시나리오 (5~10개):
  ┌──────┬──────────────────────────────────────────────────────────┐
  │ 번호 │ 시나리오                                                │
  ├──────┼──────────────────────────────────────────────────────────┤
  │  1   │ 로그인 → 메인 진입 확인                                 │
  │  2   │ 매장 스케줄 관리: 검색 → 달력 표시 확인                 │
  │  3   │ 매장 스케줄 관리: 날짜 클릭 → 상세폼 표시 → 저장       │
  │  4   │ 매장 마스터: 검색 → 그리드 데이터 표시 확인             │
  │  5   │ 쿠폰 관리: 신규 등록 → 조회 → 수정 → 삭제              │
  │  6   │ 공통코드 셀렉트박스 정상 로드 확인                      │
  └──────┴──────────────────────────────────────────────────────────┘

  이 정도면 "사이트가 기본적으로 돌아가는지" 확인에 충분하다.
  배포 후 스모크 테스트 용도.


■ ⑤ Storybook — 이 프로젝트에서는 장식
================================================================================

  Storybook이 유용한 경우:
    - 공통 UI 컴포넌트 라이브러리를 만들 때
    - 디자이너와 협업이 잦을 때
    - 컴포넌트 재사용이 많은 대규모 프론트엔드

  이 프로젝트에서 장식인 이유:
    - 관리자 BO는 정해진 레이아웃(search-form, grid, detail) 반복
    - 디자인 변경이 거의 없음
    - 컴포넌트 단위 테스트보다 "데이터가 맞나"가 더 중요


■ 권장 테스트 전략 요약
================================================================================

  비중:

    API 테스트     ████████████████████  70%  ← 핵심
    DB 검증        █████                 15%
    Playwright     ████                  10%  ← 해피패스만
    기타           ██                     5%

  도입 순서:
    1단계: Postman 컬렉션으로 주요 API 테스트 작성 (즉시 가능)
    2단계: Playwright 해피패스 5~10개 (스모크 테스트)
    3단계: CI/CD에 Newman + Playwright 연동 (자동화)

  관리자 BO CRUD는 UI도, 주요 로직도 크게 안 바뀜
  → 한번 잘 만들어두면 유지보수 거의 없이 계속 돌릴 수 있음
